# 后端功能与逻辑说明

本文档基于当前 `discover_world` 后端代码编写，覆盖服务启动、配置、接口、业务逻辑、数据模型和关键约束。数据库建表细节见 [数据库信息.md](./数据库信息.md)，媒体存储设计见 [媒体资源存储设计要点.md](./媒体资源存储设计要点.md)。

## 1. 后端整体架构

后端使用 Go + go-zero REST 服务结构。

核心目录：

```text
main.go                         服务启动入口
etc/application.yaml            主配置
etc/storage/config/default.yaml 本地对象存储密钥配置
api/discover_world.api          go-zero API 定义
internal/handler                HTTP handler 层
internal/logic                  业务逻辑层
internal/middleware             中间件
internal/svc                    依赖组装和共享服务上下文
internal/common                 通用鉴权、响应、请求解析
internal/types                  API 请求/响应结构
model                           数据库访问层
docs                            设计与功能文档
```

分层职责：

```text
handler: 解析请求、调用 logic、写响应
logic: 参数校验、鉴权判断、业务编排
model: 数据库查询和更新
svc: 初始化配置、数据库连接、模型实例、中间件
common: JWT、统一响应、错误码、JSON 工具
```

## 2. 服务启动与配置逻辑

`main.go` 默认读取 `etc/application.yaml`，也可以通过 `-f` 指定配置文件。

启动流程：

1. `config.Load` 读取主配置。
2. 自动尝试加载同目录的 `.local` 配置覆盖 COS 客户端配置。
3. 自动加载 `etc/storage/config/default.yaml` 中的本地存储密钥，并注册为 `default` 与路径别名。
4. 设置全局 `httpx.SetErrorHandler`，把业务错误统一转为 `{code,message}`。
5. 创建 go-zero REST server，启用 CORS。
6. 创建 `ServiceContext` 并注册所有路由。

主配置包含：

```text
Name / Host / Port / MaxBytes
Mysql.DataSource
CacheRedis
Auth.AccessSecret / Auth.AccessExpire
Admin.Usernames / Admin.Emails
Cos
StorageSecrets
```

注意：文档不记录实际密钥值。对象存储密钥应放在本地配置或安全配置系统中，不应提交生产密钥。

## 3. 统一响应和错误逻辑

成功响应由 `internal/common/response.Response` 输出：

```json
{
  "code": 200,
  "message": "成功",
  "data": {}
}
```

错误由 `response.StatusError` 表达，常用错误：

```text
BadRequest           400
Unauthorized         401
Forbidden            403
NotFound             404
Conflict             409
InternalServerError  500
```

handler 内部通常使用 `httpx.ErrorCtx` 写错误，main 中的错误处理器会统一转换 HTTP 状态码和响应体。

## 4. 鉴权与管理员逻辑

普通登录使用 JWT。登录成功后，后端签发 HS256 token，claims 包含：

```text
userId: 当前账号 ID
iat:    签发时间
exp:    过期时间
```

鉴权入口在 `internal/common/auth/login_user.go`。

用户解析顺序：

1. 优先从 request context 读取 `LoginUser`。
2. 其次从 go-zero JWT context claim 读取 `userId` / `uid` / `id`。
3. 如果 context 没有，再解析 `Authorization: Bearer <token>`。
4. 查找 active 且未软删除的账号。

JWT `userId` 只接受正整数，字符串、整数、JSON number 都会统一转换；非法、空值、0、小数会被拒绝。

管理员判断在 `ServiceContext.IsAdminAccount`：

```text
1. 命中 Admin.Usernames
2. 命中 Admin.Emails
3. 用户名为 admin
```

管理员接口会经过 `AdminCheckMiddleware`，中间件验证管理员身份后把登录用户写入 context，避免后续 logic 重复查询。

## 5. 账号功能

账号模块路径：

```text
internal/handler/account
internal/logic/account
model/useraccountmodel.go
model/userprofilemodel.go
```

接口：

| 方法 | 路径 | 鉴权 | 说明 |
| --- | --- | --- | --- |
| POST | `/api/account/register` | 否 | 注册账号 |
| POST | `/api/account/login` | 否 | 邮箱密码登录 |
| POST | `/api/account/detail` | 是 | 查询账号详情 |
| POST | `/api/account/update` | 是 | 更新账号与资料 |
| POST | `/api/account/avatar/upload` | 是 | 上传并设置头像 |

注册逻辑：

1. 校验 email 格式。
2. 校验密码长度不少于 8。
3. 校验两次密码一致。
4. username 为空时使用 email 前缀。
5. 校验 email 和 username 未被其他未删除账号使用。
6. bcrypt 生成密码哈希。
7. 插入 `user_account`。
8. 创建默认 `user_profile`。

登录逻辑：

1. 标准化 email。
2. 查询账号。
3. 检查账号 active、未删除、存在密码哈希。
4. bcrypt 校验密码。
5. 创建 JWT。
6. 更新 `last_login_at`。
7. 返回账号详情和 token。

账号更新逻辑：

1. 普通用户只能更新自己；管理员可通过普通接口更新其他账号。
2. username、email、password 按需更新。
3. nickname、bio 写入 `user_profile`。
4. profile 不存在时自动创建。

头像上传逻辑：

1. 复用媒体上传能力，存储用途 `usageType=avatar`，业务用途 `assetUsage=avatar`。
2. 上传成功后取返回的媒体资源 ID。
3. 写入 `user_profile.avatar_asset_id`。
4. 账号详情中的 `avatarUrl` 通过头像媒体对象和 bucket 动态生成。

## 6. 媒体资源功能

媒体模块路径：

```text
internal/handler/media
internal/logic/media
model/mediaassetmodel.go
model/mediaobjectmodel.go
model/storageprovidermodel.go
model/storagebucketmodel.go
model/tagmodel.go
model/taggingmodel.go
model/entitystatmodel.go
```

接口：

| 方法 | 路径 | 鉴权 | 说明 |
| --- | --- | --- | --- |
| POST | `/api/media/list` | 否 | 公开媒体分页列表 |
| POST | `/api/media/list/cursor` | 否 | 公开媒体游标列表 |
| POST | `/api/media/detail` | 是 | 媒体详情 |
| POST | `/api/media/upload` | 是 | multipart 上传图片 |
| POST | `/api/media/upload/url` | 是 | URL 拉取上传图片 |
| POST | `/api/media/delete` | 是 | 软删除媒体 |
| POST | `/api/media/review` | 管理员 | 审核媒体 |
| POST | `/api/admin/media/list` | 管理员 | 管理端媒体分页列表 |

### 6.1 上传逻辑

multipart 上传：

1. handler 解析 `file`、`id`、`title`、`description`、`category`、`tags`、`visibility`、`assetUsage`。
2. 保存到临时文件，最大 30MB。
3. 加载登录用户。
4. 解析图片元数据：格式、大小、宽高、主色、简化 blurHash。
5. 按 `usageType` 查找默认启用存储桶。
6. 新建或更新 `media_asset`，写入 `asset_usage`，状态先置为 `uploading`。
7. 生成对象 key，例如 `media/images/asset-1001/original.jpg`。
8. PUT 上传到 COS。
9. 如是替换上传，旧 `media_object` 标记为 deleted。
10. 插入新的 original `media_object`。
11. `media_asset.status` 更新为 `active`。
12. 替换标签关系。
13. 确保 `entity_stat` 统计行存在。
14. 返回媒体响应。

URL 上传：

1. 校验 URL scheme 只能是 http/https。
2. 30 秒超时下载远程图片。
3. Content-Type 必须是 image 开头。
4. 最大 10MB。
5. 后续流程与 multipart 上传一致。

当前上传只支持图片格式：

```text
jpg / jpeg / png / webp
```

### 6.2 查询逻辑

公开列表默认只返回：

```text
status = active
visibility = public
audit_status = approved
asset_usage = work
deleted_at is null
```

支持过滤：

```text
id / title / category / auditStatus / tags / fileSize
width / height / aspectRatio / fileExt / ownerUserId
searchText / createdAtFrom / createdAtTo
```

分页规则：

```text
普通分页默认 pageNum=1, pageSize=10, 最大 pageSize=300
游标分页默认 pageSize=30, 最大 pageSize=60
游标内容是 base64url(JSON{"id":最后一条ID})
```

列表响应构建做了批量查询优化：

1. 批量查询 original `media_object`。
2. 批量查询对象 bucket，避免按媒体条目 N+1 查询。
3. 批量查询统计、标签、作者账号、作者资料。
4. 按原列表顺序组装响应。

### 6.3 响应 URL 逻辑

媒体 URL 不直接存数据库，返回时动态生成。

URL 来源优先级：

```text
storage_bucket.cdn_domain
storage_bucket.endpoint
```

如果 host 没有 scheme，默认补 `https://`。对象 key 会逐段 URL escape。

图片变体：

| compressType | 说明 |
| --- | --- |
| 0 | 原始 URL |
| 1 | 仅大图自动压缩，根据文件大小选择最长边和质量 |
| 2 | 1200x1200 webp 缩略图 |
| 3 | 居中裁剪，必须提供 cutWidth/cutHeight |

如果媒体对象或 base URL 缺失，变体 URL 保持为空，不拼接无效 `?imageMogr2...` 参数。

### 6.4 权限逻辑

可管理媒体：

```text
资源 owner
管理员
```

可查看媒体：

```text
公开 + active + approved
或可管理该媒体
```

详情接口会增加浏览量。删除是软删除：`media_asset.status=deleted`、写入 `deleted_at`，并把关联 `media_object.status` 标为 deleted。

审核接口只允许管理员，`auditStatus` 只能是：

```text
pending / approved / rejected
```

审核信息写入 `media_asset.metadata_json`：

```text
reviewMessage
reviewerId
reviewTime
```

## 7. 个人主页功能

个人主页模块路径：

```text
internal/handler/profile
internal/logic/profile
model/postmodel.go
model/albummodel.go
model/assetlinkmodel.go
```

接口：

| 方法 | 路径 | 鉴权 | 说明 |
| --- | --- | --- | --- |
| POST | `/api/profile/post/list/cursor` | 是 | 用户动态游标列表 |
| POST | `/api/profile/featured/media/list` | 是 | 用户精选媒体 |
| POST | `/api/profile/album/list` | 是 | 用户相册分页列表 |

用户目标选择：

1. 未传 `userId` 时，默认查询当前登录用户。
2. 传了 `userId` 时，查询目标 active 账号。
3. 查询自己或管理员查询时 `includePrivate=true`。
4. 查询他人时只看公开内容。

动态列表：

1. 使用 ID 游标分页。
2. 查询目标用户动态。
3. 通过 `asset_link` 批量找到动态图片。
4. 批量构造媒体响应。
5. 批量加载 `entity_stat`。

精选媒体：

1. 查询目标用户 profile。
2. 用 `asset_link(owner_type=user_profile, link_role=featured)` 获取精选媒体 ID。
3. 只返回公开、已审核、未删除且 `asset_usage=work` 的作品媒体。
4. 批量构造媒体响应。

相册列表：

1. 按用户和可见性分页查询 album。
2. 批量统计相册内有效图片数。
3. 批量构造封面媒体响应。

## 8. 管理端功能

管理端接口都需要 JWT + `AdminCheckMiddleware`。

接口：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/admin/account/update` | 管理员更新账号、资料和状态 |
| POST | `/api/admin/media/list` | 管理端媒体列表 |

管理员更新账号：

1. 校验目标账号 ID。
2. 查询账号。
3. 可更新 username、email、password、status。
4. status 只能是 `active`、`disabled`、`deleted`。
5. 可同步更新 nickname、bio。

管理端媒体列表：

1. 默认 `auditStatus=pending`。
2. 支持与公开媒体列表相同的过滤项。
3. 不限制 public/approved，因此可用于审核后台。

## 10. 数据模型职责

核心表与 model 职责：

| 表 | 说明 |
| --- | --- |
| `user_account` | 登录账号、邮箱、密码哈希、状态 |
| `user_profile` | 昵称、头像媒体 ID、简介 |
| `storage_provider` | 存储服务商抽象 |
| `storage_bucket` | bucket、CDN、base path、用途 |
| `media_asset` | 业务媒体资源 |
| `media_object` | 对象存储中的真实文件 |
| `tag` | 标签定义 |
| `tagging` | 标签与媒体等目标的关系 |
| `entity_stat` | 浏览、点赞、收藏、评论、分享、下载统计 |
| `post` | 用户动态 |
| `album` | 用户相册 |
| `asset_link` | 媒体和动态/相册/主页精选等业务对象的关系 |

model 层做了几类定制查询：

```text
FindOneActive
FindByIDs
FindByWhere / CountByWhere
FindOriginalByAssetIDs
FindNamesByTargetIDs
ReplaceTargetTags
CountStatsByOwner
GetOverviewStats
```

批量 ID 查询会先过滤 0 并去重，减少多余 SQL 占位符和重复扫描。

## 11. 当前优化点

本轮后端优化保留现有功能和 API，主要改动：

1. JWT userId claim 校验收紧，只接受正整数，拒绝 0 和小数。
2. 媒体变体 URL 在 base URL 为空时返回空字符串，避免生成无效 `?imageMogr2...` 链接。
3. model 批量查询统一复用正整数 ID 去重逻辑，减少重复 SQL 参数。
4. 媒体列表响应批量加载 bucket，避免每条媒体单独查询存储桶。
5. 媒体原图查看/下载权限只计算一次。
6. 增加 auth、media、model 的边界测试，覆盖上述行为。
