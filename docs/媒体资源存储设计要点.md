# 媒体资源存储设计要点

## 1. 设计目标

本系统用于管理用户上传的图片、视频、音频、文档等媒体资源。媒体文件本身存储在对象存储服务中，例如腾讯云 COS、阿里云 OSS、AWS S3、MinIO 等，数据库只保存媒体资源信息、存储位置、访问规则和业务关联关系。

设计目标如下：

1. 支持图片、视频、音频、文档等多种媒体类型。
2. 支持 COS、OSS、S3、MinIO 等多种对象存储系统扩展。
3. 支持公开资源和私有资源的访问控制。
4. 支持缩略图、预览图、详情图等不同展示版本。
5. 支持动态、相册、精选、头像等业务场景复用同一套媒体资源。
6. 避免前端直接感知底层存储结构。
7. 避免数据库强绑定某一个云厂商。
8. 方便后期做 CDN、图片处理、转码、审核、分享、防盗链和数据迁移。

---

## 2. 核心设计原则

### 2.1 文件存对象存储，信息存数据库

媒体文件本身不存入数据库，而是存储到对象存储中。

数据库只保存：

* 媒体资源 ID
* 上传用户 ID
* 媒体类型
* 文件大小
* 文件宽高
* 文件 MIME 类型
* 对象存储 bucket
* 对象 key
* 存储 URI
* 访问类型
* 审核状态
* 业务状态
* 创建时间

错误做法：

```text
数据库直接保存大量二进制文件内容
```

推荐做法：

```text
文件内容：对象存储
文件信息：数据库
访问地址：后端动态生成
```

---

### 2.2 不直接把 URL 当作核心数据

数据库中不应该只保存一个完整 URL。

不推荐：

```text
https://example.cos.ap-guangzhou.myqcloud.com/media/images/xxx.jpg
```

推荐保存：

```text
bucket_id
object_key
storage_uri
```

原因是 URL 可能会变化：

* CDN 域名可能变化
* COS 域名可能变化
* 资源可能从 COS 迁移到 OSS
* 私有资源 URL 会过期
* 图片处理参数可能调整

真正稳定的是：

```text
存储服务商 + 存储桶 + object_key
```

---

### 2.3 业务资源和物理对象分离

系统中需要区分两个概念：

```text
media_asset  ：业务上的媒体资源
media_object ：对象存储中真实存在的文件
```

例如用户上传一张图片：

```text
media_asset
    表示：这是一张图片资源

media_object
    表示：这张图片在对象存储中的真实文件
```

如果后续生成了视频封面、转码文件、真实缩略图，也可以继续挂到同一个 `media_asset` 下。

示例：

```text
media_asset: 1001
    ├── media_object: original 原图
    ├── media_object: cover 视频封面
    └── media_object: transcoded 转码文件
```

如果缩略图只是通过 COS 图片处理参数动态生成，则不需要存为 `media_object`。

## 4. 对象 Key 命名规则

对象存储中的 `object_key` 不建议直接使用用户上传的原始文件名。

不推荐：

```text
张三的照片.jpg
1.jpg
2.jpg
user/1/1.jpg
```

原因：

* 容易冲突
* 容易被枚举
* 可能暴露用户信息
* 可能包含特殊字符
* 不利于后期迁移和清理

推荐规则：

```text
media/{media_type}/{yyyy}/{MM}/{dd}/{user_id}/{uuid}.{ext}
```

示例：

```text
media/images/2026/07/03/10001/8f7a2d91c3a94b1d.jpg
media/videos/2026/07/03/10001/a81c9e2f5b6d4c01.mp4
media/audios/2026/07/03/10001/b2d8c4e1a7f3.mp3
```

也可以使用 `asset_id` 组织：

```text
media/images/asset-1001/original.jpg
media/videos/asset-2001/original.mp4
media/videos/asset-2001/cover.jpg
```

推荐优先使用：

```text
media/{media_type}/asset-{asset_id}/{role}.{ext}
```

示例：

```text
media/images/asset-1001/original.jpg
media/videos/asset-2001/original.mp4
media/videos/asset-2001/cover.jpg
```

优点：

* 方便按资源清理
* 方便按资源迁移
* 方便查找某个资源下的所有对象
* 路径结构清晰

## 5. 文件角色规则

`media_object.object_role` 表示文件在媒体资源中的角色。

常见值：

```text
original     原始文件
cover        封面文件
transcoded   转码文件
attachment   附件文件
```

图片场景：

```text
original：原图
```

视频场景：

```text
original：原视频
cover：视频封面
transcoded：转码后的视频
```

文档场景：

```text
original：原始文档
cover：文档封面图
attachment：附件
```

如果缩略图、预览图只是通过云端图片处理动态生成，则不需要作为真实对象入库。

---

## 6. 图片版本规则

图片展示通常需要多个版本：

```text
thumbnail  缩略图，用于图片流、列表
preview    预览图，用于详情预览
detail     详情图，用于大图展示
avatar     头像图
original   原图
```

系统通过 `media_variant_rule` 管理不同存储服务商的图片处理参数。

例如 COS：

```text
thumbnail -> imageMogr2/thumbnail/400x
preview   -> imageMogr2/thumbnail/1200x
detail    -> imageMogr2/thumbnail/2000x
avatar    -> imageMogr2/thumbnail/200x200
```

前端不应该自己拼接任意图片处理参数。
正确做法是：

```text
前端请求媒体列表
后端根据规则生成固定版本 URL
前端直接使用后端返回的 thumbnailUrl / previewUrl
```

## 7. 是否需要存储缩略图

默认情况下，不需要为每张图片都提前存储缩略图。

推荐策略：

```text
普通图片：只存原图，缩略图通过 COS/OSS 图片处理动态生成
热门图片：异步生成真实缩略图并持久化存储
视频封面：建议真实生成并存储
```

### 7.1 不需要真实存缩略图的场景

```text
普通图片流
个人主页图片
普通相册
旅游攻略图片
访问量不高的内容
```

### 7.2 建议真实存缩略图的场景

```text
首页高频曝光图片
热门图片
头像
视频封面
需要极致加载速度的场景
需要降低图片处理费用的场景
需要跨云迁移的场景
```

推荐采用混合策略：

```text
冷门资源：动态处理
热门资源：固化缩略图
```

## 8. 访问控制规则

媒体资源分为公开资源和私有资源。

### 8.1 公开资源

公开资源可以返回 CDN URL。

适合：

```text
公开图片流
公开相册
公开动态
用户公开头像
旅游攻略封面
```

公开资源也需要防护：

```text
CDN 缓存
Referer 防盗链
访问限流
异常流量告警
必要时加水印
```

### 8.2 私有资源

私有资源不能返回永久 URL。

适合：

```text
私密相册
未审核内容
仅自己可见内容
仅粉丝可见内容
付费内容
后台审核素材
```

私有资源访问流程：

```text
1. 前端请求媒体资源
2. 后端校验当前用户权限
3. 校验通过后生成短时签名 URL
4. 前端使用签名 URL 访问资源
5. URL 过期后失效
```

签名 URL 建议有效期：

```text
私密图片预览：1 - 5 分钟
普通私有图片：5 - 15 分钟
大文件下载：15 - 60 分钟
后台审核素材：1 - 5 分钟
```

## 9. URL 返回规则

前端需要展示图片时，后端可以返回 URL，但不要暴露底层存储逻辑。

推荐返回结构：

```json
{
  "id": 1001,
  "type": "image",
  "visibility": "public",
  "width": 4000,
  "height": 3000,
  "urls": {
    "thumbnail": "https://cdn.example.com/xxx?style=thumbnail",
    "preview": "https://cdn.example.com/xxx?style=preview",
    "original": null
  },
  "permissions": {
    "canViewOriginal": false,
    "canDownload": false
  }
}
```

私有资源返回示例：

```json
{
  "id": 1002,
  "type": "image",
  "visibility": "private",
  "urls": {
    "thumbnail": "https://cdn.example.com/private/xxx?sign=abc&expires=300",
    "preview": "https://cdn.example.com/private/yyy?sign=def&expires=300",
    "original": null
  },
  "expiresIn": 300,
  "permissions": {
    "canViewOriginal": false,
    "canDownload": false
  }
}
```

无权限时：

```json
{
  "id": 1003,
  "urls": null,
  "reason": "no_permission"
}
```

## 10. 上传规则

推荐使用前端直传对象存储，后端负责签发临时凭证或预签名 URL。

上传流程：

```text
1. 前端向后端申请上传
2. 后端创建 media_upload_session
3. 后端生成 object_key
4. 后端返回临时凭证或预签名上传 URL
5. 前端直传对象存储
6. 上传完成后，对象存储回调后端，或前端通知后端
7. 后端校验上传结果
8. 后端创建 media_asset
9. 后端创建 media_object
10. 后端触发异步处理任务
```

上传会话需要记录：

```text
user_id
asset_id
provider_id
bucket_id
upload_type
upload_id
object_key
original_filename
mime_type
file_size
status
expire_at
callback_payload
```

上传状态：

```text
created      已创建
uploading    上传中
completed    已完成
failed       失败
expired      已过期
```

## 11. 文件校验规则

上传完成后，后端需要校验：

```text
文件是否真实存在
文件大小是否一致
MIME 类型是否合法
文件后缀是否合法
checksum 是否一致
上传用户是否匹配
上传会话是否过期
object_key 是否属于当前用户或当前会话
```

建议限制上传类型：

```text
图片：jpg / jpeg / png / webp
视频：mp4 / mov / webm
音频：mp3 / wav / m4a
文档：pdf
```

不要只相信前端传入的 MIME 类型，需要后端或对象存储回调进行二次校验。

## 12. 安全规则

### 12.1 不暴露敏感信息

对象 key 不应包含：

```text
手机号
邮箱
身份证
真实姓名
原始文件名
连续编号
敏感业务字段
```

### 12.2 私有资源必须鉴权

错误做法：

```text
用户知道 asset_id 就能拿到 URL
```

正确做法：

```text
后端根据当前登录用户、资源所有者、资源可见性、分享权限等进行鉴权
鉴权通过后才返回短时签名 URL
```

当前后端已启用作品媒体的 `followers` 可见性。作者本人、管理员、已关注作者的登录用户
可以查看 `visibility = followers` 的作品详情、原图权限和下载响应；未关注用户访问详情或
下载返回 403。公开媒体列表和全局搜索仍只返回 `visibility = public` 的作品。

### 12.3 限制图片处理参数

不要允许前端随便传任意图片处理参数。

错误做法：

```text
/api/media/1001/url?width=99999&format=png
```

推荐做法：

```text
/api/media/1001/url?variant=thumbnail
/api/media/1001/url?variant=preview
```

后端只允许固定版本：

```text
thumbnail
preview
detail
avatar
```

### 12.4 防盗链

公开资源建议开启：

```text
CDN Referer 防盗链
User-Agent 黑名单
IP 限流
访问频控
异常流量告警
```

但 Referer 防盗链不能替代真正的权限控制。

### 12.5 原图访问控制

原图通常比预览图更敏感。

推荐策略：

```text
普通用户默认只能看 preview
作者本人可以看 original
管理员可以看 original
下载原图需要单独权限
```

---

## 13. 删除规则

删除媒体资源时，不建议立即物理删除对象存储文件。

推荐流程：

```text
1. 用户删除资源
2. media_asset 标记 deleted
3. media_object 标记 deleted
4. 前端不再展示
5. 定时任务延迟清理对象存储文件
```

优点：

```text
支持误删恢复
避免数据库删除成功但对象存储删除失败
方便失败重试
方便审计
```

建议保留周期：

```text
普通媒体：7 - 30 天
违规媒体：根据业务规则处理
临时上传文件：1 - 24 小时
```

---

## 14. 生命周期规则

对象存储建议配置生命周期策略。

### 14.1 临时上传目录

```text
temp/
```

建议自动清理：

```text
1 - 3 天后删除
```

### 14.2 回收站目录

```text
trash/
```

建议自动清理：

```text
7 - 30 天后删除
```

### 14.3 冷数据

访问频率低的资源可以转低频存储或归档存储：

```text
STANDARD -> IA -> ARCHIVE
```

是否启用取决于业务访问频率和恢复成本。

## 15. CDN 规则

公开资源建议通过 CDN 分发，而不是直接暴露对象存储源站。

推荐访问路径：

```text
前端 -> CDN -> 对象存储
```

CDN 配置建议：

```text
开启缓存
开启 HTTPS
开启防盗链
开启访问日志
配置合理缓存时间
图片处理结果尽量缓存
```

缓存时间建议：

```text
头像：较短缓存，或 URL 加版本号
普通图片：较长缓存
热门图片：长缓存
临时签名 URL：短缓存或不缓存
```

## 16. 数据库字段规则

### 16.1 类型字段

推荐使用 `VARCHAR` 存储语义值：

```text
media_type
status
visibility
object_role
provider_type
access_type
```

示例：

```text
media_type = image
status = active
visibility = public
```

优点：

```text
可读性强
扩展方便
跨数据库友好
排查问题方便
```

代码层使用枚举进行约束。

### 16.2 时间字段

通用时间字段：

```text
created_at
updated_at
deleted_at
```

含义：

```text
created_at：创建时间
updated_at：更新时间
deleted_at：软删除时间
```

### 16.3 JSON 字段

JSON 字段用于存储非核心扩展信息：

```text
metadata_json
config_json
extra_json
callback_payload
```

不建议把高频查询字段放到 JSON 中。

## 18. 典型场景示例

### 18.1 用户上传图片

```text
1. 创建上传会话
2. 生成 object_key
3. 前端直传 COS
4. 上传完成
5. 创建 media_asset
6. 创建 media_object，role = original
7. 生成 thumbnail / preview URL 返回前端
```

### 18.2 用户发布动态并带图片

```text
1. 用户先上传图片，生成 media_asset
2. 用户发布 post
3. 使用 asset_link 关联 post 和 media_asset
4. owner_type = post
5. link_role = attachment
```

### 18.3 用户把图片加入相册

```text
1. 创建 album
2. 使用 asset_link 关联 album 和 media_asset
3. owner_type = album
4. link_role = album_item
```

### 18.4 用户设置精选图片

```text
1. 选择 media_asset
2. 使用 asset_link 关联 user_profile 和 media_asset
3. owner_type = user_profile
4. link_role = featured
```

## 19. 后端返回 DTO 规则

数据库可以拆得细，但接口不能复杂。

前端不应该直接理解：

```text
media_asset
media_object
storage_bucket
storage_provider
media_variant_rule
```

后端应该返回可直接使用的 DTO：

```json
{
  "id": 1001,
  "type": "image",
  "title": "海边日落",
  "width": 4000,
  "height": 3000,
  "aspectRatio": 1.333,
  "urls": {
    "thumbnail": "https://cdn.example.com/xxx?style=thumbnail",
    "preview": "https://cdn.example.com/xxx?style=preview"
  },
  "createdAt": "2026-07-03 10:30:00"
}
```

前端只负责展示，不负责拼接对象存储地址。

## 20. 扩展其他存储系统

系统应通过统一接口访问存储服务。

接口示例：

```java
public interface StorageClient {

    String providerType();

    UploadResult upload(UploadRequest request);

    PresignedUrlResult generateUploadUrl(PresignedUrlRequest request);

    PresignedUrlResult generateDownloadUrl(PresignedUrlRequest request);

    void delete(DeleteRequest request);

    boolean exists(ExistsRequest request);

    FileMetadata getMetadata(MetadataRequest request);
}
```

实现类：

```text
CosStorageClient
OssStorageClient
S3StorageClient
MinioStorageClient
LocalStorageClient
```

业务代码不直接依赖 COS SDK，而是依赖统一的 `StorageClient`。

扩展新存储系统时，只需要：

```text
1. 新增 StorageClient 实现
2. 新增 storage_provider 配置
3. 新增 storage_bucket 配置
4. 复用 media_asset 和 media_object
```

## 21. 总结

整体存储策略如下：

```text
文件存对象存储
信息存数据库
URL 后端动态生成
公开资源走 CDN
私有资源走短时签名 URL
缩略图优先云端动态处理
热门资源再固化缩略图
```
